Building a Scalable Database Driven Web Application

by: Christopher Shireman



One of the biggest problems when someone tries to build a web based application is that their business grows. They build the application, it works for a while, and then they out-grow it. There are steps you can take to make sure that your application can grow with your business without having to re-write it.

First, make sure you have a solid database design. Without a solid design in your database you will not be able to get your application off the ground. You need to make sure that all the database objects and relations between those objects is setup correctly.

Second, make sure that your indexes are all correct. Nothing can slow your database down like a table that hasn't been indexed properly. Look at all the ways that you will be recalling the information from your tables, make sure that the proper columns are indexed before you start coding.

Third, use property tables. These are tables that house additional information about your primary database object. This is information that you did not anticipate when you first built your database. Some designers shy away from this kind of structure, but I have found that it solves a lot of problems when you are dealing with database structures spread across multiple servers with different versions of your application runnning.

Fourth, use as few database calls as possible to get your work done. One of the biggest problems when building applications is repeating the same simple call to the database to get a single piece of information about a single object. It is far better to make one call to get all the information about all the objects you are interested in at one time. This one query might take longer than any one of the individual queries, but you won't use up nearly as many resources to get access to your database. This saves time and can really speed up your application.

Fifth, for really large applications that are used by multiple clients, segment the data out into seperate sets of tables for each client. I personally use ID numbers appended to the end of each table name to keep track of which tables belong to which client. This can end up being harder to manage, but the efficiency you gain in application performance is well worth it.

I am sure that there are many other ways that you can speed up your database applications, but these are some of my favorites. If you need more resources for building web application, check out the MySQL website at http://mysql.com. Good luck with your web applications.
